# 본스탬프 내 네이버 검색 시스템 구축 기획안

---

## 1-1. 운영 전제

- 본 시스템은 **기존 회원/예약/등록/출석 백엔드와 완전히 분리된 별도 환경**에서 운영
- 기존 서비스와 직접 연동하지 않고, 관리자가 필요한 테스터만 별도로 등록하여 관리
- 테스터 식별 정보는 최소한으로 저장
  - 예) 회원번호, 이름 또는 휴대폰 뒷자리, 소속 그룹, 활성 여부
- 관리자는 등록된 테스터에게 그룹별/주차별 키워드를 할당하고, 앱은 이 분리된 시스템에만 요청

### 테스터 저장 정보(MVP 확정)

- 테스터 기본 저장 항목
  - 회원번호
  - 이름
  - 그룹
  - 활성 여부
  - 휴대폰 뒷자리
- 이 중 **휴대폰 뒷자리는 선택값(nullable)** 으로 저장

### 앱 API 구성(초안)

- **조회 API**: 앱 사용자의 회원번호를 받아, 오늘 검색 대상인지 확인
  - 대상이 아니거나 이미 오늘 검색을 완료한 경우: **명시적 상태값을 포함한 응답** 반환
  - 대상인 경우: 오늘 수행할 검색 키워드 반환
  - 조회 응답은 **검색어 문자열 + 키워드 ID**를 포함
- **기록 API**: 앱이 검색 시작(버튼 클릭) 사실을 서버에 전송
  - 앱은 **회원번호 + 조회 API에서 받은 키워드 ID**를 서버에 전송
  - 서버는 이를 검증한 뒤 회원번호, 그룹, 키워드, 시각 등 필요한 정보를 기록
  - 기록된 데이터는 관리자 로그/통계 화면에서 조회

### API 응답 방식(확정)

- 조회 API와 기록 API는 모두 **명시적 상태값을 포함한 JSON 응답**을 사용
- 대상 아님, 비활성, 미등록, 제한 초과, 키워드 미설정 등도 단순 빈 바디가 아니라 **상태를 표현하는 응답 포맷**으로 통일
- 외부 응답 포맷은 **성공/실패만 표현하는 단순 상태값** 기준으로 유지
- 실패 사유의 세부 구분은 외부 응답에 노출하지 않고, 필요 시 **내부 로그에서만 추적**
- 일반적인 업무상 실패(미등록, 비활성, 제한 초과, 키워드 없음 등)는 **HTTP 200 + 실패 상태값**으로 응답
- 잘못된 API Key 등 인증 실패만 **401 또는 403 계열 상태코드**로 구분
- 응답 포맷은 MVP에서 다음 수준으로 단순화
  - 성공: `ok=true`
  - 실패: `ok=false`
  - 조회 성공 시에만 필요한 데이터(검색어 문자열, 키워드 ID)를 포함

### 앱 식별 및 API 접근 방식(확정)

- 앱 사용자는 **회원번호**로 식별
- 별도 로그인/세션 인증은 두지 않고, 앱이 서버에 요청할 때 **공유 API Key**를 함께 전송
- API 엔드포인트 자체는 인터넷에서 접근 가능하지만, 서버는 사전에 합의한 API Key가 없으면 요청을 거부
- 목적은 외부 제3자가 회원번호를 무작위로 넣어 호출하는 것을 막는 것
- MVP 단계에서는 앱-서버 간 단일 공유 비밀값 방식으로 운영하고, 요청 헤더 기반 인증을 기본 전제로 함
- 동일 회원이 여러 기기에서 호출하더라도 **기기 단위는 구분하지 않고 회원번호 기준으로만 처리**

### 조회 가능 여부 판정 규칙(확정)

- 기본 제한은 **하루 1회**
- 같은 회원번호로 당일 이미 검색 기록이 있으면, 조회 API는 추가 키워드를 내려주지 않음
- 추가로 **월 n회 제한**을 둠
- 월간 제한의 기본값은 **월 2회**
- `n` 값은 관리자 화면에서 수정 가능해야 함
- 즉, 운영 기본값은 `하루 1회 제한 + 관리자 설정형 월간 횟수 제한`
- 날짜 및 월간 제한 계산 기준 시간대는 **Asia/Seoul**
- 비활성 회원은 조회 대상에서 제외하며, 조회 API는 **HTTP 200 + ok=false**로 처리
- 비활성 회원에 대한 기록 API도 **HTTP 200 + ok=false**로 차단하여 외부 동작을 단순하게 유지
- 등록되지 않은 회원번호가 호출한 경우에도 조회 API는 **HTTP 200 + ok=false**로 처리
- 다만 미등록 회원 호출은 **내부 경고 로그**로 남겨 추적 가능해야 함
- 내부 경고 로그 대상은 다음을 포함
  - 미등록 회원번호 호출
  - 비활성 회원 호출
  - 잘못된 API Key 사용 요청
- 조회 API의 역할은 **현재 시점에 검색 대상인지 판정하고, 가능하면 키워드를 내려주는 것**
- 하루 1회 / 월 n회 제한은 **조회 API의 대상 판정 규칙**으로 적용

### 기록 이벤트 정의(확정)

- 서버가 기록하는 1회의 의미는 **앱이 네이버 검색을 열려고 시도한 시점**
- 서버는 실제로 네이버 화면이 성공적으로 열렸는지까지는 보장하지 않음
- 관리자 통계 화면과 로그의 기준도 **실제 검색 완료 수가 아니라 검색 실행 시도 수**
- 동일 회원이 같은 날 여러 번 기록 API를 호출하더라도, **로그는 모두 저장**
- 즉, 운영 제한과 별개로 **원시 시도 로그는 최대한 보존**하여 관리자 확인이 가능해야 함
- 기록 API에서 전달된 키워드 ID는 **해당 회원의 현재 그룹 + 현재 주차에 유효하게 할당된 키워드**이면 허용
- 즉, 조회 응답 발급 이력을 별도로 추적하지 않고 **현재 유효한 할당 기준으로 검증**
- 전달된 키워드 ID가 해당 회원의 현재 유효한 그룹/주차 조건과 맞지 않으면 **기록을 거부**
- 이 경우 **내부 경고 로그**를 함께 남겨 이상 호출을 추적
- 기록 API는 조회 API의 하루/월 제한을 다시 강제하기보다, **원시 시도 로그 보존**을 우선
- 따라서 현재 회원, 현재 그룹, 현재 주차, 현재 유효 키워드 조건이 맞는 시도는 **중복이어도 기록 가능**

### 기술 방향(결정)

- **Backend**: `FastAPI`
- **Admin Frontend**: `Jinja2 + HTMX + Alpine.js`
- **배포**: `Render`
- **DB**: `Supabase Postgres`
- **파일 스토리지**: 사용하지 않음
- **운영 시간대**: `Asia/Seoul`
- **배포 형태**: Render의 단일 웹 서비스 1개 + Supabase Postgres 연결을 기본 전제로 함

### MVP 운영 기준(확정)

- 이번 1차 결과물은 **제한된 실제 운영이 가능한 MVP**를 목표로 함
- 오픈 목표 시점은 **2026-05-29**
- 초기 운영 규모는 **약 100명**
- 품질 우선순위는 다음과 같음
  - **최우선**: 앱의 검색 대상 조회 및 클릭 기록 기능 안정성
  - **차선**: 관리자 통계 화면, CSV 다운로드, 부가 관리 편의성
  - 즉, 관리자 화면 일부가 다듬어지지 않았더라도 **조회/기록 핵심 기능이 안정적이면 MVP 범위 내 허용**

## 2. 테스터 그룹 및 키워드 할당 시스템 `Backend/Logic`

### ① 그룹 관리 및 테스터 할당

- 테스터들을 A, B, C, D 등 여러 그룹으로 생성하고 관리할 수 있는 구조
- 관리자는 개별 테스터를 특정 그룹에 자유롭게 할당/변경할 수 있어야 함
- 그룹은 테스터 속성에 붙는 단순 문자열이 아니라, **별도 테이블로 관리되는 독립 엔터티**로 설계
- 그룹은 삭제하지 않고, 필요 시 **비활성 처리만 허용**
- 테스터의 그룹 변경은 **저장 즉시 이후 조회부터 반영**

### 테스터 등록 방식(MVP 확정)

- 관리자 화면에서는 **1명씩 수기 등록**만 지원
- 초기 오픈 대상자 약 100명은 **개발자가 DB에 직접 일괄 적재**하는 방식으로 준비
- 즉, CSV 업로드 기능은 MVP 범위에서 제외
- 테스터는 삭제하지 않고, 필요 시 **비활성 처리만 허용**
- 동일 회원번호로는 **중복 재등록을 허용하지 않음**
- 기존 비활성 테스터를 다시 운영할 때는 **새 레코드 생성이 아니라 재활성화 방식**으로 관리

### ② 주차별 테스크(키워드) 매핑

- **그룹별 키워드 할당**: 특정 주차(Week)에 각 그룹이 작업할 키워드를 관리자가 미리 세팅
  - 예) 1주차 - A그룹: `"키워드1"`, B그룹: `"키워드2"` 할당
- **자동 키워드 연동**: 테스터가 앱에서 버튼을 클릭할 때, 본인이 속한 그룹에 해당 주차에 할당된 키워드가 자동으로 검색어(query)로 입력되어 네이버로 이동
- 키워드는 할당 시마다 자유 텍스트를 직접 넣는 방식이 아니라, **별도 키워드 테이블로 관리**
- 그룹별 주차 할당은 키워드 레코드를 참조하는 방식으로 설계
- 키워드는 삭제하지 않고, 필요 시 **비활성 처리만 허용**
- 키워드명은 **중복을 허용하지 않음**
- 비활성 키워드는 기존 할당 이력에는 남을 수 있으나, **조회 시 후보에서는 자동 제외**

### 주차 기준(확정)

- 주차는 **매주 월요일 시작** 기준으로 계산
- 그룹별 키워드 할당도 이 주차 기준을 따름
- 주차별 할당 데이터는 **연도+주차번호가 아니라 주차 시작일(월요일 날짜)** 기준으로 저장
- MVP에서는 **한 주차에 한 그룹당 여러 키워드 할당을 허용**
- 조회 시에는 해당 주차/그룹에 할당된 키워드 중 **랜덤으로 1개를 반환**
- 랜덤 반환 방식으로 앱과 운영 구조는 단순하게 유지하되, 같은 주차 안에서도 키워드 분산이 가능하도록 함

### 키워드 미설정 시 동작(확정)

- 회원이 조회 조건을 만족하더라도, 해당 주차의 소속 그룹에 할당된 키워드가 없으면 **HTTP 200 + ok=false**를 반환
- 별도 대체 키워드나 이전 주 키워드는 사용하지 않음

## 3. 데이터 기록 및 로깅 `Backend/Log`

### ① 개별 테스터 활동 기록

- **기록 시점**: 테스터가 앱 내에서 검색 버튼을 클릭하는 순간
- **필수 수집 항목**
  - 테스터 식별 정보 (ID / 이름 / 소속 그룹)
  - 클릭 일시 (Timestamp)
  - 당시 할당되어 클릭된 키워드명
- 로그에는 참조 ID만 두는 것이 아니라, **당시 표시값 스냅샷도 함께 저장**
  - 예) tester_id, group_id, keyword_id 와 함께 당시 회원번호, 이름, 그룹명, 키워드명 저장

### ② 테스터 운영 계획

- **권장 횟수**: 인당 월 2회 실시 권장
- **운영 로직**: 그룹별/주차별 로테이션을 통해 작업량이 분산되도록 관리

## 4. 관리자 페이지 `Admin Side`

### 관리자 인증 방식(MVP 확정)

- 관리자 페이지는 **아이디/비밀번호 기반 자체 로그인** 방식으로 보호
- 관리자 계정은 여러 개 생성할 수 있어야 함
- 다만 MVP에서는 **권한 레벨 구분 없이 모든 계정이 동일한 관리자 권한**을 가짐
- 관리자 화면에서 **관리자 계정 추가 기능**을 제공
- 관리자 계정은 삭제하지 않고, 필요 시 **비활성 처리만 허용**
- 관리자 로그인 아이디는 **단순 문자열**을 사용하며, 시스템 내에서 **유일해야 함**
- 관리자 비밀번호는 **최소 길이 검사 + 영문/숫자 포함 규칙**을 적용
- 각 관리자는 로그인 후 **자기 자신의 비밀번호만 변경 가능**
- 관리자 비밀번호는 평문 저장하지 않고 **해시 저장**을 전제로 함
- 비밀번호 찾기, 이메일 인증, 2차 인증은 MVP 범위에서 제외

### ① 실시간 작업 현황 대시보드

- **날짜별 통계**: 키워드당 일자별 총 클릭 수(작업량) 그래프 및 표
- **키워드별 통계**: 각 키워드별 누적 작업량 집계
- **테스터별 통계**: 각 테스터 인원별 참여도 및 작업 현황 (추후 보상체계 등과도 연계)
- 대시보드 기본 집계는 **원시 시도 수**와 **중복 제거된 고유 참여자 수**를 함께 제공
- MVP에서는 **시각화 그래프는 제외**하고, 숫자 요약/표/로그/필터 중심으로 구성

### ② 테스터 상세 로그 조회 및 관리

- **필터 기능**: 날짜별, 그룹별, 키워드별, 테스터별 상세 검색 결과 필터링
- **데이터 추출**: 통계 및 로그 데이터의 엑셀(CSV) 다운로드 기능
- **할당 관리**: 주차별 키워드-그룹 매핑 설정 화면
- MVP에서는 **상세 로그 CSV 다운로드만 지원**
- 통계용 CSV는 별도로 제공하지 않으며, 필요 시 상세 로그 CSV를 기준으로 후처리
- 주차별 키워드 할당 화면은 **그룹 선택 + 주차 선택 + 키워드 여러 개 선택 후 저장** 방식으로 구성
- 동일한 그룹/주차에 대해 다시 저장하면, **기존 할당 전체를 새 선택값으로 덮어씀**

## 5. MVP 구현 가정(질문 없이 진행하는 기본값)

- 앱 API 인증 헤더는 단일 공유 키 기반으로 구현
- 관리자 로그인은 세션/쿠키 기반 웹 로그인으로 구현
- 초기 데이터 적재용 시드 또는 간단한 DB 입력 절차를 제공
- 관리자 화면은 모바일 최적화보다 데스크톱 운영 효율을 우선
- 통계 화면은 카드/표 중심으로 구성하고, 복잡한 분석 기능은 제외
- 감사 로그는 우선 앱 시도 로그와 내부 경고 로그 중심으로 두고, 별도 상세 운영 감사 기능은 제외
- 구현 중 추가 선택지가 생기더라도, 위에서 확정한 원칙인 **조회/기록 안정성 우선, 관리자 기능은 단순하게**를 기준으로 판단
