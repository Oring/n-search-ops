## User Scenarios & Testing *(mandatory)*

### User Story 1 - 앱 검색 대상 조회 및 시도 기록 (Priority: P1)

본스탬프 앱은 회원번호와 공유 API Key를 사용해 오늘 검색 대상 여부를 조회하고,
대상일 때 받은 키워드 ID로 검색 시도 기록을 남길 수 있어야 한다.

**Why this priority**: 오픈 시점 MVP의 핵심 가치는 검색 대상 판정과 클릭 기록의
안정성에 있으며, 관리자 화면 일부 미완성보다 이 흐름의 정상 동작이 우선이다.

**Independent Test**: 시드된 테스터, 그룹, 키워드, 주차 할당 데이터가 있는 상태에서
조회 API와 기록 API만 호출해도 오늘 대상 판정, 랜덤 키워드 반환, 중복 시도 기록,
실패 상태 응답을 모두 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 활성 테스터가 현재 주차에 유효한 그룹 키워드를 가지고 있고 당일/당월
   제한을 넘지 않았을 때, **When** 앱이 API Key와 회원번호로 조회 API를 호출하면,
   **Then** 서버는 `ok=true`와 함께 검색어 문자열 및 키워드 ID를 반환한다.
2. **Given** 활성 테스터가 당일 이미 한 번 조회 성공 후 검색 대상이 아니게 된 상태일
   때, **When** 앱이 같은 회원번호로 다시 조회 API를 호출하면, **Then** 서버는
   HTTP 200과 `ok=false`를 반환하고 추가 키워드를 내려주지 않는다.
3. **Given** 활성 테스터가 조회로 받은 키워드 ID가 현재 그룹/주차에 여전히 유효할 때,
   **When** 앱이 기록 API를 호출하면, **Then** 서버는 시도 로그를 저장하고
   `ok=true`를 반환한다.
4. **Given** 기록 API에 전달된 키워드 ID가 현재 테스터의 그룹/주차 할당과 맞지 않을
   때, **When** 앱이 기록 API를 호출하면, **Then** 서버는 로그 저장을 거부하고
   HTTP 200과 `ok=false`를 반환하며 내부 경고 로그를 남긴다.

### User Story 2 - 운영자 데이터 관리 및 관리자 인증 (Priority: P2)

운영자는 관리자 로그인 후 테스터, 그룹, 키워드, 주차별 할당, 월간 제한값, 관리자
계정을 관리할 수 있어야 한다.

**Why this priority**: 앱 API가 실제 운영되려면 운영자가 대상자와 할당 규칙을 스스로
관리해야 하며, 관리자 화면은 인증으로 보호되어야 한다.

**Independent Test**: 관리자 계정으로 로그인한 뒤 테스터 1건 등록, 그룹 생성,
키워드 생성, 주차 할당 저장, 월간 제한 변경, 관리자 계정 추가, 본인 비밀번호 변경을
 완료하면 이 스토리의 운영 가치를 검증할 수 있다.

**Acceptance Scenarios**:

1. **Given** 활성 관리자 계정이 있을 때, **When** 운영자가 올바른 아이디와 비밀번호로
   로그인하면, **Then** 세션이 생성되고 보호된 관리자 화면에 접근할 수 있다.
2. **Given** 동일 회원번호의 비활성 테스터가 이미 존재할 때, **When** 운영자가 같은
   회원번호를 다시 등록하려고 하면, **Then** 시스템은 신규 생성 대신 기존 테스터를
   재활성화하도록 안내하거나 동일 레코드를 갱신한다.
3. **Given** 운영자가 특정 그룹과 주차를 선택한 상태일 때, **When** 여러 키워드를
   선택해 저장하면, **Then** 해당 그룹/주차의 기존 할당 전체가 새 선택값으로
   대체된다.
4. **Given** 운영자가 본인 비밀번호 변경 화면에 있을 때, **When** 영문과 숫자를
   포함한 최소 길이 기준을 충족하는 새 비밀번호를 입력하면, **Then** 시스템은
   해시된 비밀번호를 저장하고 다음 로그인부터 새 비밀번호만 허용한다.

### User Story 3 - 운영 현황 조회, 상세 로그 필터링, CSV 다운로드 (Priority: P3)

운영자는 관리자 화면에서 원시 시도 수, 고유 참여자 수, 키워드/테스터별 현황을
표 형태로 조회하고, 상세 로그를 필터링해 CSV로 내려받을 수 있어야 한다.

**Why this priority**: 운영 데이터는 중요하지만 앱 조회/기록 기능보다 우선순위가
낮으며, MVP에서는 그래프보다 표와 CSV 제공이 핵심이다.

**Independent Test**: 테스트 로그를 적재한 뒤 관리자 화면에서 날짜/그룹/키워드/테스터
필터를 조합해 결과를 조회하고, 같은 조건으로 상세 로그 CSV를 다운로드해 내용과 건수가
일치하는지 확인하면 된다.

**Acceptance Scenarios**:

1. **Given** 검색 시도 로그가 누적되어 있을 때, **When** 운영자가 대시보드에
   접속하면, **Then** 날짜별/키워드별/테스터별 요약 수치와 원시 시도 수 및 고유
   참여자 수를 표로 확인할 수 있다.
2. **Given** 운영자가 상세 로그 화면에서 날짜, 그룹, 키워드, 테스터 조건을 선택했을
   때, **When** 검색을 실행하면, **Then** 조건에 맞는 로그만 목록에 표시된다.
3. **Given** 상세 로그 필터 결과가 화면에 표시된 상태일 때, **When** 운영자가 CSV
   다운로드를 실행하면, **Then** 같은 필터가 반영된 상세 로그 CSV 파일을 받을 수
   있다.

### Edge Cases

- 미등록 회원번호, 비활성 테스터, 잘못된 API Key 요청은 외부 응답을 단순한
  `ok=false` 또는 인증 오류로 유지하되 내부 경고 로그에는 원인을 남겨야 한다.
- 현재 주차에 그룹 키워드가 하나도 없으면 조회 조건을 만족해도 키워드를 반환하지
  않는다.
- 월간 제한값이 변경된 직후에는 이후 조회부터 즉시 새 제한을 적용해야 한다.
- 같은 회원이 기록 API를 하루에 여러 번 호출해도 원시 시도 로그는 모두 저장하되,
  현재 유효한 할당이 아니면 저장하지 않는다.
- 비활성 그룹 또는 비활성 키워드는 신규 조회 후보에서 제외되지만 과거 로그 및 과거
  할당 내역의 표시값은 유지되어야 한다.
- Asia/Seoul 기준 월요일 시작 주차 계산이 자정 경계에서 틀어지지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 기존 회원/예약/출석 백엔드와 분리된 별도 운영 환경에서
  동작해야 한다.
- **FR-002**: 시스템은 앱 조회 API에서 공유 API Key와 회원번호를 받아 인증 후
  오늘 검색 대상 여부를 판정해야 한다.
- **FR-003**: 시스템은 조회 API 성공 시에만 `ok=true`, 검색어 문자열, 키워드 ID를
  포함한 JSON을 반환해야 한다.
- **FR-004**: 시스템은 미등록, 비활성, 제한 초과, 키워드 미설정 같은 업무상 실패를
  HTTP 200과 `ok=false`로 반환해야 한다.
- **FR-005**: 시스템은 잘못된 API Key 등 인증 실패에만 401 또는 403 계열 상태코드를
  사용해야 한다.
- **FR-006**: 시스템은 테스터를 회원번호, 이름, 그룹, 활성 여부, 휴대폰 뒷자리
  선택값으로 저장해야 한다.
- **FR-007**: 시스템은 동일 회원번호의 중복 재등록을 허용하지 않아야 하며, 비활성
  테스터 재사용 시 신규 레코드 대신 재활성화 방식으로 처리해야 한다.
- **FR-008**: 시스템은 그룹을 독립 엔터티로 저장하고, 삭제 대신 비활성 처리만
  허용해야 한다.
- **FR-009**: 시스템은 키워드를 독립 엔터티로 저장하고, 키워드명 중복을 허용하지
  않으며, 삭제 대신 비활성 처리만 허용해야 한다.
- **FR-010**: 시스템은 그룹별 주차 할당을 주차 시작일(월요일 날짜) 기준으로 저장하고,
  한 그룹/한 주차에 여러 키워드를 연결할 수 있어야 한다.
- **FR-011**: 시스템은 조회 시 현재 주차와 그룹에 연결된 활성 키워드 중 하나를 랜덤
  선택해 반환해야 한다.
- **FR-012**: 시스템은 조회 API에서 Asia/Seoul 기준 하루 1회 및 관리자 설정형 월간
  횟수 제한(기본값 2회)을 적용해야 한다.
- **FR-013**: 시스템은 비활성 테스터와 미등록 회원번호를 조회 대상에서 제외해야 하며,
  관련 호출을 내부 경고 로그로 남겨야 한다.
- **FR-014**: 시스템은 기록 API에서 회원번호와 키워드 ID를 받아 현재 회원, 현재 그룹,
  현재 주차, 현재 유효 키워드 조건을 만족할 때만 `ok=true`로 기록해야 한다.
- **FR-015**: 시스템은 기록 API에서 동일 회원의 중복 호출이라도 현재 유효 조건을
  만족하면 원시 시도 로그를 모두 저장해야 한다.
- **FR-016**: 시스템은 시도 로그에 참조 ID뿐 아니라 당시 회원번호, 이름, 그룹명,
  키워드명 등 표시값 스냅샷을 함께 저장해야 한다.
- **FR-017**: 시스템은 관리자 페이지를 아이디/비밀번호 기반 로그인과 세션/쿠키로
  보호해야 한다.
- **FR-018**: 시스템은 여러 관리자 계정을 지원하고, 관리자 아이디를 유일하게
  유지해야 하며, 삭제 대신 비활성 처리만 허용해야 한다.
- **FR-019**: 시스템은 관리자 비밀번호를 평문이 아닌 해시로 저장하고, 최소 길이와
  영문/숫자 포함 규칙을 검증해야 한다.
- **FR-020**: 시스템은 각 관리자가 로그인 후 자기 자신의 비밀번호만 변경할 수 있게
  해야 한다.
- **FR-021**: 시스템은 관리자 화면에서 테스터 단건 등록/수정, 그룹 관리, 키워드 관리,
  그룹-주차 할당 저장, 월간 제한값 수정 기능을 제공해야 한다.
- **FR-022**: 시스템은 주차별 할당 저장 시 동일 그룹/주차의 기존 할당 전체를 새
  선택값으로 덮어써야 한다.
- **FR-023**: 시스템은 관리자 화면에서 날짜별, 그룹별, 키워드별, 테스터별 상세 로그
  필터를 제공해야 한다.
- **FR-024**: 시스템은 관리자 화면에서 원시 시도 수와 중복 제거된 고유 참여자 수를
  포함한 요약 통계를 표 형태로 제공해야 한다.
- **FR-025**: 시스템은 상세 로그 화면의 현재 필터 결과를 CSV로 다운로드할 수 있게
  해야 한다.
- **FR-026**: 시스템은 통계용 CSV 별도 제공 없이 상세 로그 CSV만 지원해야 한다.
- **FR-027**: 시스템은 미등록 회원번호 호출, 비활성 회원 호출, 잘못된 API Key 사용,
  유효하지 않은 키워드 ID 기록 시도를 내부 경고 로그로 남겨야 한다.
- **FR-028**: 시스템은 초기 운영용 테스터/그룹/키워드/관리자 데이터를 준비할 수 있는
  시드 또는 간단한 DB 적재 절차를 제공해야 한다.
- **FR-029**: 시스템은 단일 웹 서비스와 관리형 관계형 데이터베이스 조합으로 배포 및
  운영 가능해야 한다.
- **FR-030**: 시스템은 조회 API와 기록 API의 운영 상태를 확인할 수 있는 헬스체크 또는
  이에 준하는 검증 지점을 제공해야 한다.

### Key Entities *(include if feature involves data)*

- **테스터**: 회원번호를 기준으로 식별되는 운영 대상자. 이름, 휴대폰 뒷자리, 활성 여부,
  현재 그룹을 가진다.
- **그룹**: 테스터 묶음을 표현하는 독립 엔터티. 이름과 활성 여부를 가지며 주차별
  키워드 할당의 기준이 된다.
- **키워드**: 검색어 원본 문자열을 저장하는 독립 엔터티. 이름은 유일하며 활성 여부를
  가진다.
- **주차별 그룹 키워드 할당**: 특정 주차 시작일과 특정 그룹에 대해 여러 키워드를
  연결하는 규칙 집합이다.
- **검색 시도 로그**: 앱의 기록 API 호출 결과로 저장되는 원시 시도 이벤트. 참조 ID와
  표시값 스냅샷, 호출 시각을 포함한다.
- **내부 경고 로그**: 미등록, 비활성, 인증 실패, 유효하지 않은 키워드 호출 등 운영상
  이상 징후를 추적하는 이벤트다.
- **관리자 계정**: 관리자 페이지 접근 주체. 유일한 로그인 아이디, 해시 비밀번호,
  활성 여부를 가진다.
- **운영 설정**: 월간 조회 제한 기본값과 같이 운영자가 바꿀 수 있는 전역 설정이다.

## Audit & Data Handling *(mandatory)*

- **Captured Events**: 앱 조회 성공/실패, 앱 기록 성공/실패, 미등록/비활성/API Key 오류
  경고, 관리자 로그인 성공/실패, 관리자 계정 생성/비활성화, 테스터 생성/수정/재활성화,
  그룹/키워드 생성 및 비활성화, 주차별 할당 저장, 월간 제한 변경, CSV 다운로드
- **Data Changes**: 테스터, 그룹, 키워드, 할당, 관리자 계정, 운영 설정, 검색 시도 로그,
  내부 경고 로그 레코드가 생성 또는 갱신된다. 삭제 대신 비활성화가 기본이다.
- **Retention / Reconciliation**: 원시 시도 로그와 내부 경고 로그는 운영 검증과 사후
  확인을 위해 보존되어야 하며, 관리자 화면 요약 수치는 상세 로그와 대조 가능해야 한다.
- **Export / Reporting Impact**: 관리자 상세 로그 조회, 대시보드 요약 표, 상세 로그 CSV
  다운로드가 추가된다.

## Security & Access *(mandatory)*

- **Actors / Roles**: 앱 클라이언트, 운영 관리자, 비활성 관리자, 미등록 또는 비활성
  테스터
- **Access Rules**: 앱 API는 공유 API Key가 있는 요청만 허용한다. 관리자 페이지와 CSV
  다운로드는 로그인한 활성 관리자만 사용할 수 있다. 관리자는 동일 권한을 가지며 자기
  비밀번호만 변경할 수 있다.
- **Sensitive Data**: 회원번호, 이름, 휴대폰 뒷자리, 관리자 로그인 아이디, 비밀번호
  해시, API Key, 세션 쿠키
- **Abuse / Misuse Risks**: API Key 없는 무작위 호출, 미등록 회원번호 대입 호출,
  비활성 계정 접근, 잘못된 키워드 ID를 사용한 기록 위조, 로그/CSV를 통한 개인정보
  노출. 대응으로는 헤더 기반 인증, 세션 보호, 비밀번호 정책 및 해시, 서버측 키워드
  유효성 검증, 경고 로그, 관리자 인증 강제가 필요하다.

## Operational Readiness *(mandatory)*

- **Verification Plan**: 조회/기록 API, 관리자 인증, 할당 관리, 대시보드 집계, CSV
  다운로드에 대한 자동 테스트를 작성하고, 실제 배포 환경에서는 헬스체크와 핵심 사용자
  플로우 수동 검증을 수행한다.
- **Observability**: 앱 조회/기록 요청 로그, 내부 경고 로그, 관리자 로그인 이벤트,
  배포 서비스 로그, 헬스체크 결과를 운영 확인 수단으로 사용한다.
- **Rollback / Recovery**: 스키마 변경은 마이그레이션으로 관리하고, 잘못된 배포는
  이전 Render 배포로 되돌리거나 Supabase 마이그레이션 롤백 기준을 문서화해야 한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 운영자가 준비한 활성 테스터 100명 규모에서 유효한 앱 조회 요청의 95%
  이상이 1초 이내에 명시적 JSON 응답을 반환한다.
- **SC-002**: 유효한 앱 기록 요청은 100% 원시 시도 로그로 저장되고, 잘못된 요청은
  100% `ok=false` 또는 인증 오류로 일관되게 처리된다.
- **SC-003**: 운영자는 관리자 화면에서 테스터 등록, 그룹/키워드 관리, 주차별 할당,
  월간 제한 수정 작업을 각 3분 이내에 완료할 수 있다.
- **SC-004**: 대시보드 및 상세 로그 필터 결과의 요약 건수와 CSV 다운로드 건수가
  검증 시점에 서로 일치한다.
- **SC-005**: 배포 후 핵심 사용자 플로우 검증에서 조회 API, 기록 API, 관리자 로그인,
  상세 로그 조회, CSV 다운로드가 모두 성공해야 한다.

## Assumptions

- 초기 약 100명의 오픈 대상 데이터는 개발자가 시드 또는 DB 적재 절차로 준비한다.
- 관리자 화면은 모바일 최적화보다 데스크톱 운영 효율을 우선한다.
- 기존 회원 시스템과 직접 연동하지 않고, 이 시스템 내부 테스터 데이터만 사용한다.
- 그래프 시각화, 통계 전용 CSV, 비밀번호 찾기, 이메일 인증, 2차 인증, CSV 일괄 업로드는
  MVP 범위에서 제외한다.
- 앱은 조회 API로 받은 키워드 ID를 기록 API에 그대로 전달할 수 있다.
