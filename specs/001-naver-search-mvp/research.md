# Research: 네이버 검색 운영 MVP

## 결정 1: 단일 FastAPI 서비스에 앱 API와 관리자 화면을 함께 배치한다

- **Decision**: FastAPI 하나로 JSON API, Jinja2 관리자 화면, 세션 인증, CSV 다운로드를 모두 제공한다.
- **Rationale**: MVP의 핵심은 운영 안정성이다. 별도 프런트엔드 번들 또는 복수 서비스는 배포와 롤백 복잡도만 높인다.
- **Alternatives considered**:
  - SPA 프런트엔드 + 별도 API 서버: 권한/배포/상태 관리 복잡도가 증가해 제외
  - Django 단일 앱: 가능하지만 현재 요구된 기술 방향이 FastAPI이므로 제외

## 결정 2: 데이터 접근은 SQLAlchemy 2.x + Alembic 마이그레이션으로 통일한다

- **Decision**: 운영 DB는 Supabase Postgres를 사용하고, 앱 내부 ORM은 SQLAlchemy 2.x, 스키마 변경은 Alembic 마이그레이션으로 관리한다.
- **Rationale**: Postgres 운영 환경과 SQLite 테스트 환경을 함께 지원하기 쉽고, 명시적 마이그레이션 이력으로 롤백 및 배포 추적이 가능하다.
- **Alternatives considered**:
  - SQLModel: 편하지만 세밀한 관계/제약 제어와 장기 유지보수 측면에서 SQLAlchemy 직접 사용이 더 단순
  - ORM 없이 raw SQL만 사용: 빠르지만 관리자 화면/집계/테스트 생산성이 떨어짐

## 결정 3: 검색 가능 여부 제한은 성공한 검색 시도 로그 기준으로 계산한다

- **Decision**: 조회 API의 하루 1회, 월 n회 제한은 `ok=true`로 저장된 검색 시도 로그의 일/월 건수를 기준으로 계산한다.
- **Rationale**: 요구사항에서 기록 이벤트가 실질적인 검색 시도 기준이며, 조회만 성공하고 기록이 없는 경우는 실제 수행으로 보지 않는 해석이 운영상 가장 일관된다.
- **Alternatives considered**:
  - 조회 성공 횟수 기준 제한: 조회 후 미수행 상황과 실제 시도 로그가 어긋나 운영 해석이 어려움
  - 별도 발급 토큰/조회 이력 추적: MVP 복잡도 증가

## 결정 4: 그룹-주차 키워드 할당은 정규화된 연결 테이블로 저장하고 저장 시 덮어쓴다

- **Decision**: `(group_id, week_start, keyword_id)` 유니크 조합을 갖는 연결 테이블을 두고, 운영자가 동일 그룹/주차를 저장하면 해당 범위 행을 삭제 후 새 선택값을 삽입한다.
- **Rationale**: 여러 키워드 허용, 랜덤 선택, 기존 전체 덮어쓰기 요구를 가장 단순하게 구현할 수 있다.
- **Alternatives considered**:
  - JSON 배열 저장: 필터링/제약/랜덤 쿼리/감사 추적이 불편해 제외
  - 버전드 헤더-디테일 구조: 이력에는 좋지만 MVP에서 과도함

## 결정 5: 관리자 인증은 서버 세션 + 해시 비밀번호로 구현한다

- **Decision**: 로그인 성공 시 서명된 세션 쿠키를 발급하고, 비밀번호는 강도 검증 후 해시 저장한다.
- **Rationale**: 관리자 화면이 서버 렌더링 기반이므로 세션 인증이 가장 간단하며, 별도 토큰 저장소 없이 운영할 수 있다.
- **Alternatives considered**:
  - JWT 기반 관리자 인증: 서버 렌더링 UI에는 불필요하게 복잡
  - 외부 OAuth 로그인: MVP 범위와 맞지 않음

## 결정 6: 감사성 요구를 위해 로그를 세 가지로 분리한다

- **Decision**: 검색 시도 로그, 내부 경고 로그, 관리자 감사 로그를 별도 테이블로 둔다.
- **Rationale**: 사용자 시도 데이터와 운영 이상 징후, 관리자 변경 이력을 섞지 않아도 각 목적별 조회/보존/필터링이 명확해진다.
- **Alternatives considered**:
  - 단일 범용 이벤트 테이블: 구현은 단순하지만 관리자 화면과 집계 쿼리가 복잡해짐

## 결정 7: 관리자 화면은 HTMX 중심의 부분 업데이트를 사용하되, 기본 동작은 풀 페이지로도 성립하게 만든다

- **Decision**: 필터 조회, 저장 후 목록 갱신, 플래시 메시지 갱신 등에 HTMX를 사용하고, Alpine.js는 화면 상태와 간단한 인터랙션에만 사용한다.
- **Rationale**: 요구 기술을 충족하면서도 JavaScript 복잡도를 억제할 수 있다.
- **Alternatives considered**:
  - Alpine.js만으로 전부 처리: 서버 응답 분할과 HTML 재사용성이 떨어짐
  - HTMX 없이 순수 폼 전환만 사용: 가능하지만 운영 UX가 다소 둔해짐

## 결정 8: 배포와 운영 검증은 헬스체크, 앱 API 스모크 테스트, 관리자 수동 시나리오를 조합한다

- **Decision**: `/health` 엔드포인트, 자동 테스트, 배포 후 실제 API 호출 및 관리자 화면 검증을 모두 수행한다.
- **Rationale**: 조회/기록 안정성 우선 원칙에 맞게 운영 전 확인 단계를 명시적으로 남길 수 있다.
- **Alternatives considered**:
  - 단순 헬스체크만 사용: 핵심 요구 검증이 부족함
  - E2E 브라우저 자동화만 사용: 초기 구현 비용이 커서 MVP 대비 과함
